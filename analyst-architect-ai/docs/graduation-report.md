# Отчёт по выпускному проекту

## AnalystGuru — AI System Analyst Copilot

> Курс: Вайб-кодинг | Варианты: **2 (ИИ-рецензент ТЗ) + 5 (Система знаний команды)**

---

## Название проекта

**AnalystGuru** — AI-помощник системного аналитика и архитектора с ролевой моделью и авторизацией. Объединяет **вариант 2** (AI-рецензент технических заданий) и **вариант 5** (система знаний команды с памятью и RAG-поиском) в единый enterprise-продукт.

---

## Ценность (для кого и зачем)

**Для кого:** системные и бизнес-аналитики, архитекторы ПО, тимлиды и менеджеры, новые сотрудники.

**Проблема:**
1. Ручная рецензия одного ТЗ занимает **2–4 часа**
2. Знания команды раскиданы по файлам и головам людей — поиск занимает 15–30 минут
3. Аналитики делают вид, что уверены, даже когда данных недостаточно
4. Нет разграничения доступа: любой может менять настройки AI или удалять данные

**Решение:** AnalystGuru превращает текст ТЗ в структурированный отчёт за 5–10 минут, отвечает на вопросы по базе знаний с цитатами-источниками, честно говорит "данных недостаточно", и разграничивает доступ по трём ролям с обязательной авторизацией.

---

## 3 сценария пользователя

### Сценарий 1 — Рецензия ТЗ "в один клик" (вариант 2)
Аналитик входит по логину/паролю, вставляет текст ТЗ, нажимает "Рецензия" и через 20 секунд получает структурированный отчёт.

### Сценарий 2 — Поиск по базе знаний с источниками (вариант 5)
Новый сотрудник задаёт вопрос в AnalystGuru и получает точный ответ с цитатой из внутреннего документа — без звонков коллегам.

### Сценарий 3 — Ролевой доступ и честная неопределённость
Аналитик пытается открыть настройки AI-провайдеров и получает `403 Forbidden` — эта функция доступна только архитекторам и администраторам. При этом при рецензии расплывчатого ТЗ система честно ставит флаг `needs_review=true` вместо выдумывания ответа.

---

## Ролевая модель

| Роль | Логин по умолчанию | Права |
|------|--------------------|-------|
| **Администратор** (`admin`) | admin / admin123 | Всё + управление пользователями |
| **Архитектор** (`architect`) | architect / architect123 | Всё бизнес-функции + настройки AI |
| **Аналитик** (`analyst`) | analyst / analyst123 | Документы, рецензии, KB, память, аудит |

Авторизация: JWT (HS256, 8 часов), пароли — bcrypt-хеш. Роли проверяются на backend через `Depends(require_analyst/require_architect/require_admin)` на уровне роутеров — невозможно забыть защитить новый эндпоинт.

## AI-операция

### Схема строгого JSON (AI-рецензия)

```json
{
  "summary": "2–6 предложений",
  "risks": [{"severity": "low|medium|high", "description": "..."}],
  "missing_requirements": ["..."],
  "questions_to_client": ["..."],
  "acceptance_criteria": ["..."],
  "confidence": "high|medium|low",
  "needs_review": false
}
```

### Параметры LLM

| Параметр | Значение | Обоснование |
|---------|---------|-------------|
| `temperature` | 0.2 | Минимальная вариативность → предсказуемый JSON |
| `max_tokens` | 4096 | Достаточно для полной рецензии |

### Когда нужна ручная проверка

| Условие | Код причины |
|---------|------------|
| Документ < 8 слов | `TOO_VAGUE_INPUT` |
| Противоречия в требованиях | `CONTRADICTORY_INPUT` |
| Ошибка парсинга JSON | `INVALID_JSON` |
| Нет источников в KB | `NO_SOURCES_FOUND` |

## Память/контекст

5 типов памяти (semantic, episodic, decision, risk, requirement) автоматически подмешиваются в промпт AI-рецензии — топ-5 по каждому типу — что делает рецензии учитывающими накопленный опыт команды.

---

## Контроль качества

1. Валидация входа: длина текста (10–30 000 символов), обязательные поля
2. Валидация выхода: `json.loads()` → `Pydantic(ReviewSchema)` → эвристика противоречий → `safe_fallback_review()` при любой ошибке
3. Правило KB: `sources == [] → needs_review = True`
4. **Новое:** Валидация авторизации — каждый защищённый эндпоинт проверяет JWT и роль до выполнения бизнес-логики

## Риски и меры снижения (обновлено с учётом авторизации)

| # | Риск | Мера снижения |
|---|------|---------------|
| 1 | LLM возвращает невалидный JSON | `safe_fallback_review()` — безопасный ответ без падения |
| 2 | Утечка API ключей | `.gitignore`, маскирование ключей в UI |
| 3 | Галлюцинации LLM | `needs_review=true` при низкой уверенности |
| 4 | Недоступность LLM API | Поддержка 3 провайдеров, переключение в 1 клик |
| 5 | **Несанкционированный доступ к данным** | JWT-авторизация обязательна для всех бизнес-эндпоинтов (401 без токена) |
| 6 | **Аналитик меняет настройки AI без прав** | RBAC на уровне роутеров: `require_architect` (403 для analyst) |
| 7 | Брутфорс авторизации | bcrypt-хеширование, JWT с TTL 8 часов |
| 8 | Утечка пароля через переиспользование | Возможность сброса пароля администратором |
| 9 | Потеря данных SQLite | Ежедневный бэкап, миграция на PostgreSQL |
| 10 | Неверная рецензия влияет на решение | Чёткий флаг ⚠️, принцип "AI помощник, не замена" |

## 3 строки для портфолио

**Проблема:** Системные аналитики тратят 2–4 часа на рецензию ТЗ и 15–30 минут на поиск ответа в документах команды, при этом любой сотрудник может менять критичные настройки без контроля доступа.

**Решение:** AnalystGuru — AI-копилот с ролевой моделью (admin/analyst/architect) и JWT-авторизацией: рецензирует ТЗ за 10 минут, отвечает по базе знаний с цитатами, честно сигнализирует о неопределённости, разграничивает доступ к настройкам и управлению пользователями.

**Результат:** FastAPI + React 18 + SQLite + JWT/bcrypt + Anthropic/OpenAI/ProxyAPI; 39 автотестов (pytest, включая RBAC), Docker Compose, полный аудит-журнал, 200+ i18n-ключей (RU/EN), руководства пользователя и администратора с C4/UML диаграммами.

## Примеры curl-запросов

```bash
# 1. Авторизация
TOKEN=$(curl -s -X POST http://localhost:8000/auth/login \
  -d "username=analyst&password=analyst123" | python3 -c "import sys,json;print(json.load(sys.stdin)['access_token'])")

# 2. Создать документ (требует JWT)
curl -X POST http://localhost:8000/documents \
  -H "Authorization: Bearer $TOKEN" -H "Content-Type: application/json" \
  -d '{"title":"Форма заявок","text":"Нужна форма заявки для клиентов с полями имя, email, телефон."}'

# 3. Запустить рецензию
curl -X POST http://localhost:8000/documents/{ID}/review -H "Authorization: Bearer $TOKEN"

# 4. Прямой AI-вызов
curl -X POST http://localhost:8000/ai/review \
  -H "Authorization: Bearer $TOKEN" -H "Content-Type: application/json" \
  -d '{"text":"Сделать сайт."}'

# 5. Задать вопрос в KB
curl -X POST http://localhost:8000/kb/ask \
  -H "Authorization: Bearer $TOKEN" -H "Content-Type: application/json" \
  -d '{"question":"Какой SLA установлен на ответы?"}'

# 6. Попытка доступа без токена — 401
curl -i http://localhost:8000/documents

# 7. Аналитик пытается зайти в настройки — 403
curl -i -H "Authorization: Bearer $TOKEN" http://localhost:8000/settings/providers
```

## Сложности и как решал

| Сложность | Решение |
|-----------|---------|
| LLM возвращает JSON в Markdown-блоках | `extract_json()` — стриппинг фенсов |
| bcrypt v4 несовместим с passlib | Зафиксирована версия `bcrypt==4.0.1` |
| pytest не видел таблицы (DB race condition) | Autouse sync fixture создаёт таблицы + сидирует пользователей до старта тестов |
| Защита роутеров сломала старые тесты | Обновлены фикстуры: `auth_headers` логинится как analyst перед каждым защищённым запросом |
| Два axios-инстанса (defaults vs create()) не делились токеном | Добавлен request-интерцептор в `api/index.ts`, читающий токен из localStorage |
