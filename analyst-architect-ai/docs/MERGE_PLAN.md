## 4. Новый модуль: Экономика и окупаемость (флагманская фича)

### Идея

Каждый документ/ТЗ, загруженный в систему, можно привязать к **build_project** — "проекту-приложению", которое создаётся с помощью AnalystGuru. Для этого проекта система:

1. **AI-декомпозиция** (`task_estimates`): по тексту ТЗ/URS/SRS AI разбивает работу на задачи с story points → переводит в часы по ролям (backend/frontend/QA/DevOps) через настраиваемый коэффициент.
2. **Экономическая оценка** (`economic_estimates`): считает CAPEX (стоимость разработки = часы × ставки ролей), OPEX (хостинг + LLM-токены + поддержка/мес), выгоду (экономия времени сотрудников от использования нового приложения × их ставка), и производные метрики: ROI %, payback period (мес), NPV на горизонте 12/24/36 мес.
3. **Факт vs план** (`economic_actuals`): после внедрения можно занести фактические цифры (реальные часы, реальная экономия) — система показывает отклонение план/факт.
4. **Экспорт бизнес-кейса**: DOCX/PDF отчёт "Экономическое обоснование проекта" — то же самое, что аналитик обычно готовит вручную в Excel/Word для питча руководству.

### Строгий JSON контракт AI-декомпозиции задач

```json
{
  "tasks": [
    {"name": "string", "role": "backend|frontend|qa|devops|analyst", "story_points": 1, "estimated_hours": 4.0, "risk_multiplier": 1.0}
  ],
  "total_hours_by_role": {"backend": 120, "frontend": 80, "qa": 40, "devops": 16},
  "confidence": "high|medium|low",
  "needs_review": false
}
```

### Формула экономики (прозрачная, без "чёрного ящика")

```
CAPEX = Σ(hours_by_role × hourly_rate_by_role)
OPEX_monthly = hosting_cost + (llm_calls_per_month × avg_tokens × price_per_token) + support_hours × hourly_rate
Benefit_monthly = time_saved_hours_per_month × avg_employee_rate
Payback_months = CAPEX / (Benefit_monthly − OPEX_monthly)
ROI_12m_% = ((Benefit_monthly − OPEX_monthly) × 12 − CAPEX) / CAPEX × 100
```

Все ставки (`hourly_rate_by_role`, `avg_employee_rate`, `hosting_cost`, `price_per_token`) — настраиваются в UI, дефолты берутся из рыночных значений РФ 2026.

### Новый экран Frontend: "💰 Экономика"

- Список build-проектов с бейджами: `payback: 2.3 мес`, `ROI: 340%`
- Карточка проекта: слайдеры ставок → живой пересчёт ROI (без похода на backend, но с кнопкой "Сохранить в БД")
- График CAPEX/OPEX/Benefit по месяцам (recharts) с точкой break-even
- Таблица план/факт с цветовой индикацией отклонений

## 5. Прочие новые возможности (помимо экономики)

| Функция | Обоснование |
|---|---|
| **Reasoning-режимы (CoT/ReAct)** | Перенос из GitHub-репо — повышает прозрачность решений архитектора |
| **Risk Catalog API + UI** | Модель уже была, но без API — открываем CRUD, чтобы аналитик пополнял библиотеку типовых рисков |
| **Project Lessons API + UI** | Аналогично — библиотека ретроспектив, доступная для поиска |
| **Dashboard** | Единый экран для менеджера: сколько рецензий, экономия часов, среднее ROI по всем build-проектам |
| **Seed-examples API** | Быстрый онбординг демо-данных одной кнопкой в UI (Settings → Demo data) |
| **Alembic миграции** | Профессиональный процесс апгрейда схемы БД в продакшене |
| **OpenRouter как 4-й провайдер** | Больше гибкости в выборе моделей (Llama, Mistral, Gemini через один API) |
| **Экспорт бизнес-кейса в DOCX/PDF** | Готовый артефакт для презентации руководству — закрывает "мини-экономику" из задания курса |
| **Batch-режим рецензии** | Загрузить N документов → получить N рецензий асинхронно (roadmap v1.1) |

## 7. Как это закрывает задание курса (варианты 2 + 5)

Экономический модуль — прямое развитие обязательного раздела отчёта "Мини-экономика" из задания: вместо разового расчёта "на салфетке" в тексте отчёта, система теперь **считает это как фичу продукта** для каждого нового проекта, который аналитик заводит через AnalystGuru. Это резко усиливает заявленную ценность "продукт, которому можно доверять" — теперь доверие распространяется и на финансовое обоснование, а не только на качество рецензии.
